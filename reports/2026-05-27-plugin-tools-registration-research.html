结论

datasource 不适合只靠 skill。当前 Kimi Code 代码已经支持声明式 command-backed plugin tools： manifest 负责声明 schema 和命令，tool 被模型调用时才起子进程执行。它还没有支持 import 插件 JS/TS runtime，也没有 trusted runtime handler。

当前已支持

`plugin.json` 声明工具，Kimi Code 解析后注册为 plugin__<pluginId>__<toolName>，调用时起子进程。

保留 MCP

长驻服务、多工具、OAuth、复杂集成继续走 MCP。简单数据查询、一次性计算、官方工具包可以走 command tool。

当前未支持

不 import 第三方 JS/TS 插件 runtime；不支持 runtime handler；不支持插件动态 registerTool。

现在最接近可发的形状：skills 负责教模型什么时候用，tools 负责稳定执行，manifest 负责声明 schema 和命令。

三类路线

路线 代表 核心机制 优点 主要风险
声明式 command tool Kimi CLI manifest 声明 command，调用 tool 时起子进程，stdin JSON / stdout result。 语言无关；适合 datasource；不需要加载插件代码才能发现 tool。 依赖本机 runtime；tool 调用就是本地代码执行。
skills / commands + MCP Claude Code、Codex 插件贡献 instructions、commands、skills；真实外部工具主要通过 MCP server 暴露。 边界清楚；工具协议统一；适合复杂集成。 简单官方工具也要包 MCP，成本偏高；MCP 启用/审批 UX 要做好。
in-process runtime registerTool OpenCode、PI、OpenClaw 宿主 import 插件 JS/TS runtime，插件调用 SDK 注册 tool/hook/provider。 能力最强；支持动态工具、UI、hook、provider、配置。 加载阶段就执行第三方代码；信任面最大；SDK 兼容成本高。

各家实现展开

Kimi CLI command

  • PluginToolSpec 包含 name、description、command、parameters。
  • agent 初始化时扫描 ~/.kimi/plugins，把插件 tool 加入 KimiToolset。
  • 执行时用 create_subprocess_exec，不走 shell 字符串。
  • 参数通过 stdin JSON 传入，stdout 是结果，stderr/非 0 exit 是错误。
  • 调用前走 approval；超时默认 120 秒。

Claude Code mcp

  • manifest 合并 metadata、hooks、commands、agents、skills、MCP、LSP、settings、userConfig。

  • 真实外部工具主要通过 plugin MCP server 提供。

Codex mcp

  • .codex-plugin/plugin.json 字段主要是 skills、MCP servers、apps、hooks、interface。

OpenCode runtime

  • 插件是 JS/TS runtime，宿主 resolve 之后动态 import。
  • Hooks 可以直接贡献 tool，也能拦截 permission、tool before/after、shell env。
  • tool definition 用 Zod 参数 schema 和 execute(args, context)。
  • 能力强，但加载插件本身就是执行第三方代码。

PI runtime

  • extension 通过 registerTool 注册工具。
  • tool definition 包含 name、label、description、promptSnippet、parameters、execute、renderCall/renderResult。

对 Kimi Code 的落点

当前 Kimi Code 已经把 command-backed plugin tools 接进了 core、SDK 和 TUI，但还没有 runtime plugin。

当前已经支持

  • 本地目录和 Zip URL 安装都会复制到 $KIMI_CODE_HOME/plugins/managed/<id>。
  • manifest 支持根目录 plugin.json 和 .kimi-plugin/plugin.json。
  • 支持 skills、root SKILL.md fallback、sessionStart.skill、skillInstructions。
  • 支持 tools：object 或 array 形式，inputSchema/parameters，stdin: json|none，timeoutMs。
  • tool 暴露名为 plugin__<plugin>__<tool>，默认 agent/coder profile 已包含 plugin__*。
  • 支持插件声明 mcpServers，但需要用户显式启用。

当前还不支持

  • 不支持 runtime.entry / handler 形式的 JS/TS runtime plugin。
  • 不支持插件代码动态调用 registerTool。
  • 不支持 per-plugin/per-tool enable 开关；目前 plugin tools 随插件启用进入新 session。
  • 不支持 OpenClaw 式 optional tool allowlist；当前只有 profile 里的 plugin__* gating。
  • 不支持旧 Kimi CLI 的 config_file/inject 写配置文件机制。
  • command 必须解析到插件根目录内的文件；不能直接写 node 这种 PATH command。

Runtime 插件取舍

如果只为 datasource，当前 command-backed tools 已经够用。如果还希望官方 marketplace 插件能用 JS/TS 直接实现 handler，可以做一个更窄的 trusted lazy runtime handler，但不要把它做成通用第三方 registerTool SDK。

可以接受的简单 runtime

  • 只允许官方 marketplace / trusted 插件启用。
  • manifest 仍然声明全部 tool 名称、description、schema。
  • runtime 只提供 handler 实现，不能动态新增 tool。
  • lazy import：只有 tool 真被调用时才 import runtime entry。
  • 不在 install、list、info、session start 阶段执行 runtime 代码。

不建议第一步做的 runtime

  • 不做 OpenCode/PI 式 in-process plugin SDK。
  • 不开放任意 api.registerTool。
  • 不开放 hooks、providers、commands、MCP 动态注册。
  • 不自动安装第三方 npm 依赖。
  • 不承诺外部插件 SDK 长期兼容。

复杂度判断
  • 代码复杂度：中等。需要 manifest schema、tool executor、runtime import/cache、trusted 判断、错误处理、测试。
  • 产品复杂度：中等偏高。需要定义“trusted”的来源、marketplace 签名/来源策略、runtime 插件如何在 UI 中提示风险。
  • 兼容复杂度：可控。只要明确这是 official trusted runtime handler，不是公开通用 SDK，就不需要一开始做完整 API 稳定承诺。
推荐顺序：先用现有 command-backed tools 发 datasource；如果官方 marketplace 需要更顺滑的 JS 实现，再单独加 trusted lazy runtime handler。

建议方案

当前已实现的是声明式 plugin command tools。它比 skills 里提示 Bash 更稳定，比 in-process 插件 runtime 更克制，也能支撑 datasource。下面是推荐继续使用的 datasource manifest 形状。 

{
  "name": "kimi-datasource",
  "version": "0.1.0",
  "description": "Finance datasource tools for Kimi Code.",
  "skills": "./skills",
  "tools": {
    "query_finance": {
      "description": "Query finance data.",
      "inputSchema": {
        "type": "object",
        "properties": {
          "symbol": { "type": "string" }
        },
        "required": ["symbol"],
        "additionalProperties": false
      },
      "command": "./bin/query-finance.js",
      "args": [],
      "stdin": "json",
      "timeoutMs": 120000,
      "optional": false
    }
  }
}
1. 安装

插件文件复制进 ~/.kimi-code/plugins/managed。

2. 解析

读取 plugin.json，校验 tool schema 和路径安全。

3. 注册

新 session 注册 plugin__id__tool。

4. 调用

模型调用 tool，Kimi Code 发起 approval。

5. 执行

spawn 插件内命令，stdin JSON，stdout result。

建议新增的核心字段
  • tools：object keyed by local tool name，避免数组里重复 name。
  • inputSchema：JSON Schema，直接作为模型 tool parameters。
  • command：必须是 ./ 开头的插件内相对路径。
  • args：字面量 argv，不做 shell 拼接。
  • stdin：第一版只支持 json。
  • timeoutMs：默认值可取 120000。
  • optional：后续用于 allowlist；第一版可以先解析展示。

风险控制

风险 控制方式 为什么这样做
任意代码执行 安装和 session start 不执行插件代码；只有 tool 被调用时才起子进程。 降低自动加载插件的风险，但承认 tool 调用本身是代码执行。
shell 注入 不用 shell 字符串，只用 argv spawn；参数走 stdin JSON。 避免模型参数被拼进命令行。
路径逃逸 command 必须相对插件根目录，resolve 后不能逃出 root。 避免插件 manifest 指向任意系统路径。
工具名冲突 对模型暴露 plugin__<plugin>__<tool>。 对齐 MCP 的命名空间思想，避免覆盖 builtin/MCP tools。
环境缺失 在 /plugins info 做诊断；官方插件尽量用随包 JS 或声明 runtime。 datasource 不应假设用户有 Python/pip。
危险副作用 默认走 approval；后续加 per-tool policy、allowlist、official trusted 标记。 不能把插件 tool 当成 builtin 安全能力。
最小实现原则：先让 datasource 这类插件能稳定贡献 tool，再逐步补 per-tool enable、approval policy、official trust、runtime 诊断。
